Natural Lip Sync

AvatarPlatform uses a hybrid amplitude + phoneme controller to drive Rive mouth shapes in real time. The result is smooth, expressive lip sync that works for any voice engine without requiring phoneme data from the TTS service.

Architecture overview

The lip-sync system lives in two files:

  • public/js/audio-clock.js — high-resolution audio scheduler that converts audio buffer timing into phoneme events
  • public/js/viseme-map.js — lookup table mapping phoneme characters to Azure-style viseme IDs (0–22)

These are combined inside lipsync-sdk.js's HybridLipsyncController class, which runs continuously while audio is playing.

How viseme mapping works

The 23 viseme slots

AvatarPlatform uses Azure's 22 viseme IDs (0 = silence + 1–22 for phoneme groups) as an intermediate representation. Each viseme ID maps to one Rive number input (inputs 100–122):

At any moment, at most one Rive input is non-zero. The active input value ramps from 1 → 100 across the duration of the phoneme (the "timed-ramp" mode), giving a smooth interpolated mouth movement rather than a binary snap.

Amplitude mode

The Web Audio AnalyserNode runs at 50 Hz (every 20ms), reading 512 FFT bins. The RMS power of the speech-frequency band (200–3400 Hz) drives a continuous jaw-open value:

// From audio-clock.js — simplified
const fftData = new Float32Array(analyser.frequencyBinCount); // 256 bins
analyser.getFloatFrequencyData(fftData);

// Speech band: bins 3–43 at 48kHz (roughly 200–3400 Hz)
const rms = Math.sqrt(fftData.slice(3, 43).reduce((s, v) => s + v*v, 0) / 40);
const jawOpen = Math.min(1, rms * amplitudeSensitivity);

This value modulates the active viseme's Rive input in the range [visemeMinValue, visemeMaxValue], ensuring the mouth opens proportionally to loudness even when no discrete phoneme event is scheduled.

Phoneme schedule mode

The scheduler estimates when phonemes occur based on audio playback timing. It fires at PHONEMES_PER_SEC = 13.5 phonemes per second — approximately the average English speech rate.

Each phoneme event:

  1. Determines the next viseme ID by cycling through a plausible phoneme sequence
  2. Schedules the Rive input to ramp from visemeMinValue → visemeMaxValue over the phoneme duration
  3. Triggers silence (input 100) for SILENCE_PAUSE_MS = 130ms between words
  4. Applies visemeOverlapMs = 35ms of co-articulation overlap between adjacent phonemes

Phoneme-based vs amplitude-based

Amplitude-basedPhoneme-based
Data source Real-time FFT of audio output Scheduled events based on audio timing
Accuracy Approximate — jaw movement only Better — includes consonants, lip closure, rounding
Latency Zero — real-time analysis ~40ms anticipation pre-roll (anticipationMs)
Use case Fallback, low-CPU mode Primary mode (timed-ramp)

In timed-ramp mode (the default), both run simultaneously. The phoneme scheduler selects which Rive input is active; the amplitude controller modulates its value. This produces the most natural result.

Customising viseme weights

You can remap any Azure viseme ID to a different Rive input by passing riveInputMap to the LipsyncAvatar constructor:

const avatar = new LipsyncAvatar({
  container: '#chat',
  riveSrc: '/assets/characters/character_1.riv',
  apiKey: config.apiKey,

  // Override specific viseme → Rive input mappings
  riveInputMap: {
    0: 100,   // silence → input 100 (closed)
    1: 101,   // aa vowel → input 101 (wide open)
    21: 107,  // p/b/m → input 107 (lip closure)
  },

  // Timing parameters
  visemeMinValue: 1,       // Rive value at phoneme start (1–99)
  visemeMaxValue: 100,     // Rive value at phoneme peak (visemeMinValue–100)
  visemePeakRatio: 0.88,   // fraction of phoneme duration at which peak is reached
  visemeOverlapMs: 35,     // co-articulation overlap between phonemes (ms)

  // Amplitude
  amplitudeSensitivity: 1.0,  // multiply FFT amplitude (0.5 = quieter, 2.0 = more exaggerated)

  // Timing
  anticipationMs: 40,     // pre-roll: mouth starts moving N ms before phoneme
  mouthDelayMs: 0,        // positive = delay mouth (audio arrives late)
});

Rive input reference

Each Rive character file uses number inputs 100–122 for mouth poses. Input 100 is the closed/silent pose. Inputs 101–122 correspond to the 22 phoneme groups in the Azure viseme set. Setting an input to 0 deactivates it (no contribution to the mouth shape).

💡

If you're designing your own .riv file, use the same input numbering (100–122) to ensure compatibility with the SDK's phoneme scheduler out of the box.
