Troubleshooting

Solutions to the most common issues when deploying AvatarPlatform embeds, setting up the server, and configuring voice chat.

Widget & Embed

🔴 Avatar doesn't appear on the page

Most common cause: PUBLIC_GEMINI_API_KEY (or GEMINI_API_KEY) is missing on the server. The embed config endpoint returns an empty apiKey, causing the SDK to abort before rendering.

Fix: Set at least GEMINI_API_KEY in your .env file and restart the server:

GEMINI_API_KEY=AIza...   # required
PUBLIC_GEMINI_API_KEY=AIza...  # recommended (separate restricted key)

Also check:

  • The data-public-id attribute on the script tag matches a real project's Public ID (visible in the dashboard Embed tab)
  • The script src points to the correct server URL
  • No Content Security Policy (CSP) header is blocking the script or the Rive CDN
🔇 No audio — the character's mouth doesn't move

Cause: Browsers block autoplay of audio until the user has interacted with the page. The Gemini Live WebSocket audio context is suspended on load.

Fix: The SDK automatically resumes the AudioContext on the first user gesture (click, tap, or keypress) anywhere on the page. Make sure the user clicks something — including the widget launcher — before speaking.

If audio still doesn't play after a click:

  • Check the browser console for AudioContext.resume() failed errors
  • On iOS Safari, ensure the user tapped the launcher (not just tapped elsewhere) — iOS requires the gesture to be on an audio-related element
  • Some browser extensions (ad-blockers, privacy tools) block WebAudio. Test in an incognito window.
🌐 CORS errors in the browser console

AvatarPlatform's embed routes (/embed/*) allow all origins by default — Access-Control-Allow-Origin: *. This is intentional: the embed widget runs on the visitor's website, not your own domain.

If you're seeing CORS errors:

  • The request is going to the wrong URL (e.g., localhost:3000 instead of your server)
  • A reverse proxy (nginx, Cloudflare) is stripping CORS headers — check your proxy config
  • You have a cors({ origin: 'specific-domain.com' }) override in server.js — the default is cors() (open)

The dashboard API routes (/api/*) also allow all origins. If you want to restrict these, add an origin option to the cors() call in server.js.

📱 Embedding fails on iOS Safari — no microphone access

iOS Safari requires the AudioContext to be created and resumed synchronously in response to a user gesture. The SDK creates the context on the first mic button tap, which satisfies this requirement.

Known issues on iOS:

  • iOS 15 and earlier: WebRTC mic access may fail in WKWebView (used in React Native). Use iOS 16+ or Safari on device for full voice support.
  • Low Power Mode: iOS can throttle requestAnimationFrame to 30fps, causing choppy Rive animations. This is a platform limitation.
  • Autoplay policy: Audio won't play until a tap occurs. This is by design — do not call audioContext.resume() before a user gesture.

For React Native on iOS: Ensure NSMicrophoneUsageDescription is set in Info.plist. See the React Native SDK docs.

👄 Lip sync is out of sync — mouth moves after the speech

This is caused by audio buffer latency — the audio is scheduled in the Web Audio API ahead of the current time, but the lip-sync controller reads from the analyser in real-time.

Fix: Adjust mouthDelayMs to compensate:

// If mouth moves ~100ms AFTER speech, set a negative delay
const avatar = new LipsyncAvatar({
  ...
  mouthDelayMs: -80,  // advance mouth by 80ms relative to audio
});

Typical values range from -50 to -150 depending on the audio buffer size and device. Start with -80 and tune from there.

Also try: Increasing anticipationMs (default 40ms) — this pre-rolls the mouth before the scheduled phoneme starts.

Server & Database

🗄️ Server crashes with "relation does not exist"

The Postgres schema hasn't been applied. Run supabase/schema.sql in your Supabase SQL Editor (or via psql).

psql $DATABASE_URL -f supabase/schema.sql

Also ensure the vector extension is enabled — the schema creates it with CREATE EXTENSION IF NOT EXISTS vector but Supabase Free tier has it available by default.

🔐 JWT errors — "Invalid or expired token"

This happens when JWT_SECRET changes between sessions. Tokens are signed with this secret — if the secret changes, all existing tokens become invalid and users are logged out.

Fix: Set a stable, long-lived JWT_SECRET in your environment. Never regenerate it in production unless you intend to invalidate all sessions:

# Generate a secure secret:
node -e "console.log(require('crypto').randomBytes(48).toString('hex'))"
⚡ Embeddings are slow or timing out

The Gemini Embedding API has rate limits. If you're uploading many files at once, the embedding service will queue requests and may hit the rate limit.

Symptoms: Files stuck in processing status for more than 2 minutes; server logs show [embed] 429 Too Many Requests.

Fix: Upload files one at a time, or add a delay between uploads. The embedding service (backend/services/embed.js) uses exponential backoff on 429 errors but a hard 3-retry limit.

For bulk imports, reduce chunkSize in backend/services/process.js (default 1200 chars) to create smaller chunks and reduce the number of API calls per file.

💳 Stripe webhook returns 400 "signature verify failed"

The Stripe webhook endpoint requires the raw request body for signature verification. If a middleware (body-parser, nginx gzip) modifies the body before it reaches the handler, the signature check fails.

The endpoint is already mounted before express.json() in server.js — this is correct. The issue is usually an nginx reverse proxy applying gzip or a Cloudflare transform rule.

Ensure your reverse proxy passes the body unchanged to the Node process for the path POST /api/billing/webhook.

Still stuck?

Check the server logs first — almost every error is logged with a [route/action] prefix. If you can't find the cause, book a 30-minute setup call or email support@avatarplatform.ai.
